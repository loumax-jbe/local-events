# Local Events Board

A pipeline that pulls upcoming events near a location from multiple
sources, classifies them, de-duplicates them, and shows them on a
static dashboard.

Two ways to run it:
- **Locally on your own computer** — run the scraper via cron, view the
  dashboard on your machine only (sections 1-9 below).
- **Hosted for free, as a public site anyone can visit** — GitHub runs
  the scraper on a schedule and publishes the dashboard automatically,
  no computer of yours needs to stay on (section 10).

```
config.yaml           <- location, API keys, which sources to use
setup.py                <- interactive wizard: town, travel time, event types
scraper.py               <- run this daily; pulls + stores + exports events
normalize.py              <- shared event schema + de-dup logic
classify.py                <- event_type + scale classification logic
geocode.py                  <- town name -> coordinates (used by setup.py)
runlog.py                    <- collects per-source run status (ok/empty/error/etc.)
manual_events.yaml             <- quick-add file for Instagram/word-of-mouth events
.github/workflows/
  update-events.yml              <- runs the scraper + publishes the site automatically
sources/
  ticketmaster.py               <- Ticketmaster Discovery API
  seatgeek.py                     <- SeatGeek API
  ics_calendar.py                  <- school/library/town iCal feeds
  custom_html.py                    <- config-driven scraper for sites without a feed
  manual.py                          <- reads manual_events.yaml
dashboard/
  index.html                           <- open this in a browser to view events
  status.html                            <- hidden page: what was scraped, what's missing/broken
  events.json                              <- generated by scraper.py, dashboard reads this
  sources_status.json                        <- generated by scraper.py, status.html reads this
data/
  events.db                                   <- SQLite database (created on first run)
```

## 1. Setup

```bash
cd event-scraper
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

## 2. Get free API keys

- **Ticketmaster** (concerts, sports, theater, huge coverage):
  https://developer.ticketmaster.com/ → "Get Your API Key". Free tier,
  5,000 calls/day, approved instantly.
- **SeatGeek** (secondary source, catches things Ticketmaster misses):
  https://seatgeek.com/account/develop → generates a `client_id`
  instantly, no approval wait.

Paste both into `config.yaml` under `api_keys:`. Both sources are
optional — the pipeline runs fine with just the ICS and manual sources
if you'd rather skip signing up for these.

## 3. Tell it where you are, how far you'll travel, and what you want to see

```bash
python3 setup.py
```

This asks four questions and writes the answers into `config.yaml`:

1. **Your town** — geocoded automatically (no need to look up coordinates
   yourself). Confirms the match before saving.
2. **How far you're willing to travel**, in minutes by car — converted
   into a search radius for Ticketmaster/SeatGeek. This is an
   approximation (a speed estimate, not real turn-by-turn routing), so
   it'll be roughly right rather than precise. Adjust `avg_speed_mph` in
   `config.yaml` if 30 mph is a poor fit for your area.
3. **How far ahead to look for events** — a preset menu (2 weeks, 1
   month, 2 months, 3 months, 6 months) plus a custom option if none of
   those fit. Written as `days_ahead`.
4. **Which event types you want** — a checklist (Concert, Theater &
   Performing Arts, Comedy, Family & Kids, Festival & Fair, Sports,
   Community & Civic, School & Youth, Film, Other). Press Enter to keep
   everything.

Re-run `setup.py` any time to change your answers — it only touches
`location`, `days_ahead`, and `event_type_filter`, everything else (API
keys, feed URLs, custom sites) is left alone. You can also skip the
wizard and edit `config.yaml` directly if you prefer.

## 4. Add school/library/town calendar feeds

This is usually the highest-value source for genuinely local events —
school plays, library programs, town meetings — since these essentially
never show up on Ticketmaster or SeatGeek.

Most school district, library, and municipal sites publish an iCalendar
(.ics) feed, even when it's not obvious. Look for a "Subscribe" / "RSS" /
"ICS" / "Add to Calendar" link or calendar icon near any calendar widget
on the site — very common on Finalsite-powered school sites and on
Communico/libnet.info-powered library sites. For a public Google
Calendar, use its "Secret address in iCal format" from that calendar's
settings.

`config.yaml` comes with placeholders for two real examples:

```yaml
ics_calendars:
  feeds:
    - name: "Montgomery High School (MTSD)"
      url: "PASTE_ICS_FEED_URL_HERE"
      page_url: "https://mhs.mtsd.k12.nj.us/calendar"
      category: "School"

    - name: "Princeton Public Library"
      url: "PASTE_ICS_FEED_URL_HERE"
      page_url: "https://princetonlibrary.libnet.info/events"
      category: "Library"
```

To fill in the school one: go to https://mhs.mtsd.k12.nj.us/calendar,
click "Calendar RSS Feeds" above the calendar, hover an ICAL button, and
choose "Get Standard iCal URL". Paste that into `url:`.

To fill in the library one: go to
https://princetonlibrary.libnet.info/events (apply whatever filters you
want — event type, date range — the feed will match what's showing),
look for an "Add to Calendar" or calendar-icon button near the top of
the list, choose the iCal/subscribe option, and copy the generated URL.
Worth knowing: that site is a JavaScript app with no event data in the
raw page HTML, so it's a case where `custom_html`'s CSS-selector
scraping genuinely can't work — the calendar feed is the only real
option, which happens to also be the more durable one.

Add more feeds the same way for other schools, libraries, or your
town's site — look for the same "Subscribe" / "RSS" / "Add to Calendar"
pattern first, before reaching for `custom_html` below.

## 5. Add local venues without a feed (optional)

For venues with an events page but no ICS feed, `custom_html` scrapes
it directly using CSS selectors:

```yaml
sites:
  - name: "Some Local Venue"
    url: "https://example.com/events"
    event_selector: ".event-card"      # repeating card wrapping one event
    title_selector: ".event-card__title"
    date_selector: ".event-card__date"
    link_selector: "a"
    link_attr: "href"
    scale: "Mid-size"                  # Major / Mid-size / Local-Community
```

To find the right selectors: open the venue's events page, right-click an
event listing → **Inspect**, and look for the class name that wraps each
individual event. This is the part most likely to need occasional
maintenance — if a venue redesigns their site, that one entry stops
returning events until you update its selectors. Nothing else breaks.
Prefer an ICS feed over this whenever the site has one — it won't break
on a redesign.

## 6. Add events you heard about on Instagram

The events most worth knowing about — a school musical, a library
concert — are often only posted on Instagram, with no feed or API
anywhere. Automated Instagram scraping isn't something this pipeline
does: it requires bypassing login/anti-bot systems and violates Meta's
terms, and Instagram's own Graph API only exposes accounts you manage
yourself, not ones you just want to follow.

Instead, `manual_events.yaml` is a quick-add file:

```yaml
events:
  - title: "Annie Jr."
    date: "2026-07-24T19:00"
    venue: "Montgomery High School"
    category: "School Musical"
    event_type: "School & Youth"
    source_note: "Instagram"
```

Takes about 10 seconds per event, and it flows through the exact same
dedup, classification, and dashboard display as every automated source.

## 7. Run it

```bash
python3 scraper.py
```

This prints how many events each source found, dedupes overlaps, saves
everything to `data/events.db`, and writes `dashboard/events.json`.

## 8. View the dashboard

Browsers block a local page from loading a local JSON file directly
(`file://` CORS restriction), so serve the folder instead of double-clicking
the HTML file:

```bash
cd dashboard
python3 -m http.server 8000
```

Then open **http://localhost:8000** in a browser. You'll see event cards
with search, scale, and type filtering.

## 9. Schedule it to run daily

**Mac/Linux (cron):**

```bash
crontab -e
```

Add a line to run at 6am daily (adjust paths):

```
0 6 * * * cd /full/path/to/event-scraper && /full/path/to/event-scraper/venv/bin/python3 scraper.py >> logs/scraper.log 2>&1
```

**Windows (Task Scheduler):**
Create a new task, trigger "Daily" at your preferred time, action = start a
program: `python.exe`, arguments: `scraper.py`, "start in":
the `event-scraper` folder.

The dashboard always reads whatever `events.json` currently contains, so
once the cron job is running you just reopen the page (or leave the local
server running) and it'll reflect the latest daily pull.

## Event classification

Every event gets two independent labels so the dashboard can tell a
touring arena act apart from a school play, even when both say "Concert":

- **event_type** — what it is: Concert, Theater & Performing Arts,
  Comedy, Family & Kids, Festival & Fair, Sports, Community & Civic,
  School & Youth, Film, Other. Determined by keyword matching against
  the title/venue/category (see `classify.py`), falling back to the
  source's own genre/segment data when available.
- **scale** — how big it is: **Major** / **Mid-size** / **Local-Community**.
  This is the axis that separates an arena tour from a library concert.
  - ICS calendar feeds and manual entries are Local-Community by
    definition — if it's coming from a school/library/town calendar or
    something you personally heard about locally, it's local.
  - SeatGeek events use SeatGeek's own popularity score.
  - Ticketmaster events use top ticket price as a proxy, since that API
    doesn't expose a popularity figure — tune the cutoffs in
    `config.yaml` under `classification:` if "Major" is catching too
    much or too little for your area.
  - Custom HTML venues get a per-site default (`scale:` in config),
    since a venue like a regional performing-arts center isn't
    automatically "local" the way a library calendar feed is.

Filter by both on the dashboard. Adjust the keyword rules in
`classify.py` if a particular event type keeps getting mis-sorted.

## Checking what's been scraped (and what's missing)

Every run records a status entry for every source — not just "ticketmaster:
40 events" but per-feed/per-site detail ("McCarter Theatre: 12 events",
"Montgomery HS: not configured — still a placeholder URL"). This is how
you spot a broken selector, an unfilled API key, or a feed that's quietly
returning nothing, without reading console output.

Open `dashboard/status.html` directly (it's not linked from the main
board — deliberately out of the way, not something a visitor to the
public dashboard needs to see) after running `scraper.py`. It lists every
configured source with:

- **ok** — found events
- **empty** — ran fine, zero events (worth a look — could be genuinely
  quiet, or a sign something's off)
- **error** — failed to run, with the actual error message
- **not configured** — enabled in config but missing a key/URL/file
  (this is the most common "missing source" case — a placeholder that
  was never filled in)
- **disabled** — `enabled: false` in config.yaml

Sorted with the things most worth your attention at the top: errors and
unconfigured sources first, then quiet ones, then healthy ones last.

- **Travel time is an approximation.** `setup.py` converts your answer
  into a straight-line search radius using an assumed average road
  speed, not real driving-time routing. It's a reasonable proxy (and the
  same kind of radius parameter Ticketmaster/SeatGeek's own APIs expect
  natively) but will occasionally over- or under-reach compared to an
  actual drive. Adjust `avg_speed_mph` in `config.yaml` if it's
  consistently off for your area.
- **Geocoding** uses OpenStreetMap's free Nominatim API and caches
  results in `data/geocode_cache.json` so repeat setup runs don't
  re-query the same town.

## Notes / things worth knowing

- **Dedup** matches on title + venue + date, so the same concert appearing
  on both Ticketmaster and SeatGeek collapses into one card (keeping
  whichever entry has more complete info).
- **Past events auto-prune** from the database each run.
- **Keyword filter**: set `keyword_filter` in config.yaml (e.g.
  `["music", "blues", "jazz"]`) if you only want a subset of events —
  leave it empty to keep everything.
- **Rate limits**: both ticketing APIs' free tiers comfortably cover one
  run a day for a single metro area.
- This was built and logic-tested (classification, dedup, storage, JSON
  export, dashboard rendering, and the manual quick-add) with synthetic
  sample data, including a simulated side-by-side of a major touring act
  and a small community concert to confirm they classify differently.
  The Ticketmaster and SeatGeek API integrations follow their documented
  request/response formats, but I couldn't make live calls to them or to
  a real ICS feed in this environment — run `scraper.py` with your real
  keys and feed URLs and check the console output on first use; if a
  field looks off, the fix is almost always a one-line tweak in that
  source's `_to_event()` / `_fetch_feed()` function.

## 10. Deploying as a public website (GitHub Pages)

This runs the whole pipeline on GitHub's own servers on a schedule and
publishes the dashboard at a public URL — nobody's personal computer
needs to stay on, and nobody visiting the site needs any of this
software themselves. Everything below can be done entirely through
GitHub's website — no terminal, no git command line required (though if
you're comfortable with git, that works too).

**Before you start:** a public GitHub repo means the *code* is visible
to anyone — including `config.yaml` (your town, travel radius, and any
ICS feed URLs you've added, like a school calendar link). None of that
is truly sensitive, but worth knowing since the whole point here is
that it's public. API keys are the one thing that stay private (see
step 3) — never paste a real key directly into config.yaml on a public
repo.

1. **Create the repo.** Go to https://github.com/new, name it something
   like `local-events`, set it to **Public**, and click "Create
   repository." (Free GitHub account required if you don't have one —
   sign up at github.com.)

2. **Upload the project files.** On your new repo's page, click "Add
   file" -> "Upload files," then drag in everything from this project
   folder (keeping the folder structure — `sources/`, `dashboard/`, and
   `.github/workflows/` need to stay as folders, not get flattened).
   Commit the upload.

3. **Add your API keys as secrets** (not in config.yaml): go to the
   repo's **Settings** tab -> **Secrets and variables** -> **Actions** ->
   "New repository secret." Add two:
   - Name: `TICKETMASTER_API_KEY`, value: your real Ticketmaster key
   - Name: `SEATGEEK_CLIENT_ID`, value: your real SeatGeek client ID

   These get injected automatically when the workflow runs and are
   never visible in the repo's files.

4. **Turn on Pages.** Repo **Settings** -> **Pages** -> under "Build and
   deployment," set **Source** to **GitHub Actions**.

5. **Run it the first time.** Go to the **Actions** tab, click "Update
   events and deploy" in the left sidebar, then click "Run workflow" ->
   "Run workflow." It'll take a minute or two — you can watch it run
   live by clicking into it.

6. **Visit your site.** Once that run finishes (a green checkmark),
   your dashboard is live at `https://<your-username>.github.io/<repo-name>/`.
   Share that link with anyone.

**After that, it stays current on its own** — the workflow runs every
day automatically (11:00 UTC by default; edit the `cron:` line in
`.github/workflows/update-events.yml` to change the time — the format is
minute, hour, day, month, weekday, all in UTC).

**Making changes later — no terminal needed:** click any file on
github.com (e.g. `config.yaml` or `manual_events.yaml`), click the
pencil/edit icon, make your change, and commit it directly in the
browser. That commit automatically triggers a fresh scrape and
redeploy — the live site updates within a couple minutes, no need to
manually re-run anything.

To see what was actually scraped on the last run (the hidden status
page — see "Checking what's been scraped" above): visit
`https://<your-username>.github.io/<repo-name>/status.html`.
